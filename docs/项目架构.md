## 1. 项目概述

xueli-core 是 xueli 对话框架的 Rust 标准库 crate，提供完整的 AI 对话管线。不绑定任何特定平台，下游通过实现 `PlatformAdapter` trait 接入不同消息渠道。

### 1.1 核心设计目标

- **智能对话规划** — 判断"该不该回、什么时候回、怎么回"
- **长期记忆系统** — 三层记忆（person_fact / chat_summary / conversation_recall），支持动态遗忘和语义联想
- **自主情绪引擎** — 由 Planner 输出的结构化信号驱动，按会话独立隔离
- **平台解耦** — 通过 trait 抽象，不绑定 QQ/NapCat/API
- **Trait 多态** — 所有可替换组件通过 trait 抽象，下游可替换任意组件

### 1.2 与原项目的关系

| 维度 | xueli-qq-bot (Python) | xueli-core (Rust) |
|------|----------------------|-------------------|
| 定位 | 可运行 QQ Bot 应用 | 可复用的核心库 crate |
| 平台适配器 | 内置 NapCat / API | 通过 `PlatformAdapter` trait 由下游实现 |
| WebUI | Django 控制台 | 不做（保留在 Python 侧） |
| 插件系统 | 热加载 Python 插件 | 通过 trait 暴露钩子点 |
| 许可 | MIT | MIT |

## 3. 主消息处理链

```
InboundEvent                          # 平台归一化入站事件
  │
  ▼
TimingGate                            # 时机判断：reply / wait / ignore
  │ (reply)
  ▼
Planner                               # 回复规划 + PromptPlan + 情绪信号
  │
  ▼
ContextBuilder                        # 上下文构造：对话历史 + 记忆检索 + 角色卡
  │
  ▼
ReplyAgent                            # Tool-calling 工具循环 → 生成回复
  │
  ▼
ReplyAction → PlatformAdapter.send_action()  # 平台发送
  │
  ▼
MemoryFlowService                     # 回复后副作用（记忆写入、信号更新）
```

### 3.1 处理流程要点

- **@mention / 私聊**：跳过 TimingGate 缓冲，直接进入 reply
- **群聊非 @**：消息先入 GroupMessageCollector 缓冲，攒够阈值后触发 TimingGate
- **Wait 机制**：TimingGate 返回 `wait` 时，进入等待窗口，窗口内新消息积累后重新评估
- **中断协调**：新消息可打断正在进行的推理（@/私聊不参与中断限制）
- **分段发送**：LLM 通过 `ReplyAction.segments` 控制分段，未传时按空行兜底分段

## 5. 关键约束

1. **私聊和群聊共用一条 conversation 主链**，不分裂两套逻辑
2. 回复后副作用走 `MemoryFlowService`，不在 ReplyAgent 内
3. 命名用 `conversation_*`、`platform_*` 等中性命名
4. 会话永不过期，重启后从历史存储恢复
5. 上下文窗口由 `TokenCounter` 基于 token 预算统一管理
6. 所有可替换组件通过 `src/traits/` 中的 trait 抽象
7. 平台适配器留给下游实现，core 不被平台细节污染
8. 用户侧异常静默失败，不发解释性文字
9. 语义信号失败时不伪造，直接不注入该部分提示词

## 2. MessageHandler — 高层消息编排

`MessageHandler` 是消息处理的顶层编排器，协调所有处理组件。对应 Python 版 `xueli/src/handlers/message_handler.py`。

### 2.1 组件依赖

```rust
pub struct MessageHandler<A: AIClient, P: PlatformAdapter> {
    config: Arc<XueliConfig>,
    timing_gate: DefaultTimingGate,
    planner: ConversationPlanner,
    context_builder: ConversationContextBuilder,
    reply_agent: ReplyAgent<A>,
    platform: Arc<P>,
}
```

### 2.2 处理流程

```rust
async fn handle(&self, event: &InboundEvent) -> Result<Option<ReplyAction>, String> {
    // 1. TimingGate: 判断是否回复
    let decision = self.timing_gate.should_reply(&ctx).await?;
    if decision != TimingDecision::Reply { return Ok(None); }

    // 2. Planner: 规划回复策略
    let plan = self.planner.plan(event).await?;

    // 3. ContextBuilder: 构建上下文
    let ctx = self.context_builder.build(event, &conversation, &plan).await?;

    // 4. ReplyAgent: 生成回复
    let reply = self.reply_agent.generate_reply(event, &ctx, &plan).await?;

    Ok(Some(reply))
}
```

## 3. TimingGate — 时机判断

群聊轻量 Planner：判断是否参与对话，并产出即时规划信号。对应 Python 版 `xueli/src/handlers/timing_gate.py`。

### 3.1 决策类型

```rust
pub enum TimingDecision {
    Reply,   // 立即回复
    Wait,    // 等待更多消息
    Ignore,  // 不参与
}
```

### 3.2 决策策略

- **配置了 `group_reply_decision` 模型**：调用 LLM 进行时机判断
  - 输入：窗口消息、最近历史、情绪状态、关系摘要
  - 输出：`action`（reply/wait/ignore）+ `planning_signals`
- **未配置模型**：退回规则路径（通常只在被 @ 时回复）

### 3.3 决策缓存

- 使用 `SignalCache` 缓存决策结果
- TTL：12 秒
- 缓存键：session_key + 消息摘要

### 3.4 当前状态

- ✅ LLM 调用集成（加载 `timing_gate.prompt` 模板）
- ✅ 决策缓存逻辑（`SignalCache` + TTL 12s）
- ✅ fallback 规则路径（@mention 时直接 reply）
- ✅ `planning_signals` 结构化输出解析

## 4. ConversationPlanner — 回复规划器

在主路径中负责 `plan_reply()`：输出 `PromptPlan`、`reply_reference`、当前话题 `narrative_signal` 和即时回复适配 `reply_adaptation_signal`。对应 Python 版 `xueli/src/handlers/conversation/planner.py`。

### 4.1 核心流程

```
1. 构建 system prompt（从 planner.prompt 模板加载）
2. 构建 user prompt（窗口消息 + 历史 + 上下文）
3. 调用规划模型获取决策
4. 解析 JSON 输出 → PromptPlan + reply_reference
5. @mention / 私聊 直接 reply，群聊非 @ 默认 wait
```

### 4.2 PromptPlan 结构

```rust
pub struct PromptPlan {
    pub reply_goal: String,          // continue / satisfy / cool_down / clarify
    pub continuity_mode: String,     // direct_continue / new_topic / ...
    pub timeline_detail: String,     // summary / detailed / minimal
    pub context_profile: String,     // standard / compact / expanded
    pub memory_profile: String,      // relevant / none / extensive
    pub tone_profile: String,        // balanced / warm / formal / casual
    pub initiative: String,          // gentle_follow / active_lead / ...
    pub expression_profile: String,  // plain / expressive / minimal
    pub policy: PromptSectionPolicy, // 各区块开关策略
    pub notes: String,
    pub emoji_should_send: bool,
    pub emoji_tone: String,
    pub emoji_emotion: String,
    pub emoji_intent: String,
}
```

### 4.3 当前状态

- ✅ LLM 调用集成（加载 `planner.prompt` 模板）
- ✅ JSON 输出解析 + fallback
- ✅ `reply_adaptation_signal` 输出（trust_level、formality_distance、emotional_safety）
- ✅ Planner 失败时回退默认 `PromptPlan`

## 6. ConversationContextBuilder — 上下文构建器

构建消息处理所需的完整上下文。对应 Python 版 `xueli/src/handlers/conversation/context_builder.py`。

### 6.1 构建的上下文内容

| 上下文区块 | 说明 |
|-----------|------|
| 近期对话历史 | 从 conversation store 获取，按 token 预算截断 |
| 记忆检索结果 | 人物事实、持久记忆、精确召回、动态记忆 |
| 时间上下文 | 消息间隔、会话间隔、连续性提示 |
| 视觉分析 | 图片理解结果（如有） |
| 规划信号 | TimingGate 输出的 planning_signals |
| 角色卡快照 | CharacterCardService 提供的当前快照 |
| 风格指南 | ReplyStylePolicy 生成的最终风格指南 |
| 会话恢复 | 上次会话摘要（如有） |

### 6.2 上下文窗口管理

- 主事实源：conversation store 中的 `group_messages`
- 由 `TokenCounter` 基于 `context_token_budget_ratio` 统一管理
- `max_context_length` 降级为兼容性兜底

## 7. ReplyAgent — 回复代理

Tool-calling Agent，负责回复决策与生成。对应 Python 版 `xueli/src/handlers/reply/agent.py`。

### 7.1 工具循环

每轮最多运行 `MAX_INTERNAL_ROUNDS` 次 tool-calling 循环：

```
工具列表：
  - reply          → 结束工具，输出最终回复
  - query_memory   → 检索记忆
  - query_person   → 查询人物信息
  - view_message   → 查看历史消息
  - tool_search    → 外部工具搜索（插件扩展）
```

### 7.2 执行流程

```
1. 加载 system prompt 模板（reply_agent_system_base.prompt + scene_guidance）
2. 构建 user prompt（上下文 + 记忆 + 角色卡 + 风格指南）
3. 调用 AI → 解析 tool_call
4. 执行工具 → 将结果注入下一轮消息
5. 收到 reply 工具调用 → 提取回复文本 → 分段 → 返回
```

### 7.3 插件工具扩展

- 通过 `load_plugin_tools()` 注册下游自定义工具
- 工具定义格式：`(name, handler_fn, tool_schema_dict)`

### 7.4 当前状态

- ✅ Prompt 模板加载（`PromptTemplateLoader`）
- ✅ Tool-calling 循环（工具执行 + 结果注入）
- ✅ 分段输出（`ReplyAction.segments`）
